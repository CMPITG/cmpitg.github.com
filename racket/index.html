Redirecting standard output

According to The Racket Guide, default ports could be retrieved with (current-input-port), (current-output-port), and (current-error-port). They are actually parameters, and thus could be parameterized:

#lang racket

(define (display-to-string)
  (let ([output-port (open-output-string)])
    (parameterize ([current-output-port output-port])
      (display "Hello World"))
    (get-output-string output-port)))

(display (format "Result: ~a" (display-to-string)))

New method

Thanks to suggestion from Takikawa-san (asumu on IRC):

And you can run racket command without any argument to execute the main submodule:

Old method

Suppose we have this simple Racket module:

#lang racket

(provide main)

(define (main)
  (displayln "Â¡Hola mundo!"))

Reading racket --help, one might be tempted to use:

racket -m a-module.rkt

to run the main function, which fails, and it seems unexpected. However, the Racket doc shows that for Racket to process -m, you need to explicitly tell Racket to require and do so with -t:

racket -tm a-module.rkt

Personally, I consider it as a bad design decision, but this is how itâ€™s supposed to work.

Pitfalls

copy-directory/files copies directory contents, not the directory itself

Demonstration:

ls /tmp/hello/
# -> hello.js hello.css

(copy-directory/files "/tmp/hello/" "/tmp/world/")

ls /tmp/world/
# -> hello.js hello.css

But we actually expect the hello/ directory is copied itself, not its contents. To fix this, create /tmp/world/hello/ first then do the copy:

(make-directory* "/tmp/world/hello")
(copy-directory/files "/tmp/hello/" "/tmp/world/")

And weâ€™re gonna have our desire behavior:

ls /tmp/world/
# -> hello
ls /tmp/world/hello/
# -> hello.js hello.css
